# PhageFlow v0.3 Analysis Roadmap

This roadmap ranks future additions after the v0.2 software-facing validation work. The goal is to add real analytical value while keeping PhageFlow reproducible, local-first, and conservative about biological interpretation.

## Recommendation

Do not copy whole pipelines into PhageFlow. Use established phage tools and pipelines as references for expected outputs, then wrap selected tools as optional, versioned modules with clear inputs, local database paths, provenance, report summaries, and validator checks.

## Priority 1: Better Optional-Tool Summaries

These additions are worth doing first because the workflow already has optional execution paths or artifact-summary utilities.

| Addition | Why it matters | Implementation shape |
| --- | --- | --- |
| CheckV summary parser | Implemented | `optional_tool_summary.tsv` reports quality-summary artifact status, shape, size, and checksum; `--expect-checkv-summary` validates report rows. |
| Pharokka summary parser | Implemented | `optional_tool_summary.tsv` reports annotation artifact status, file count, primary artifact class, shape, size, and checksum. |
| geNomad summary parser | Implemented | `optional_tool_summary.tsv` reports classifier artifact status, primary summary/log artifact class, shape, size, and checksum. |
| Phold artifact summary | Implemented | `optional_tool_summary.tsv`, `structural-summary`, `summarize`, and `package` expose structural-artifact availability without annotation values. |
| Clinker report summary | Implemented for HTML/alignment artifacts | `optional_tool_summary.tsv` reports synteny HTML/input-list artifact status and checksums. Static export remains future polish. |

## Priority 2: Local Reference Context

Reference context should be local-input-only.

| Addition | Why it matters | Implementation shape |
| --- | --- | --- |
| Local reference dereplication | Helps compare a new isolate against a curated local panel. | User supplies reference FASTA and metadata; output nearest-neighbor metrics and duplicate flags. |
| Local taxonomy/reference summary | Useful for cohort triage. | Summarize user-provided metadata categories and distance metrics without live database fetching. |
| Reference package manifest | Improves reproducibility. | Hash all user-provided reference files and record versions/metadata schema. |

Avoid live NCBI or public database discovery in the core workflow because results change over time and make validation harder.

## Priority 3: Host Prediction As Optional Evidence

Host prediction can be useful, but it must be clearly labeled as probabilistic.

| Addition | Why it matters | Implementation shape |
| --- | --- | --- |
| iPHoP optional wrapper | Adds established host-prediction evidence. | `--run_iphop true` with local database path; summarize top-level confidence fields and provenance. |
| Spacer-match report integration | Existing spacer matching can be more visible. | Add report table and validator checks for spacer-match summaries. |
| Host-context evidence matrix | Separates linked metadata, composition similarity, spacer evidence, and prediction evidence. | Add a host-specific claim/evidence table with limitations. |

## Priority 4: Expanded Comparative Reporting

These are useful after optional-tool summaries are stable.

| Addition | Why it matters | Implementation shape |
| --- | --- | --- |
| Pangenome method concordance in report | Makes sensitivity analysis more visible. | Import completed-run `pangenome-sensitivity` TSV into report/package. |
| Marker-tree QC dashboard | Helps audit marker coverage and topology consistency. | Add report cards from marker provenance and topology tables. |
| Synteny block summary | Adds cohort-level gene-order evidence. | Summarize block counts and artifact availability; avoid functional claims. |

## Defer Until Separate Input Contracts Exist

| Addition | Reason to defer |
| --- | --- |
| Raw-read termini and packaging inference | Requires read inputs, coverage assumptions, and separate validation data. |
| Metagenomic viral-contig discovery | Changes PhageFlow from already-sequenced genome analysis into discovery/contig-triage. |
| Full automated biological conclusion generator | Too likely to overclaim from optional artifacts. |
| Host-range, engineering, design, or wet-lab recommendation logic | Outside software workflow validation scope. |

## Candidate Tool References

Use these as external tools or design references, not as code to copy wholesale:

- CheckV for viral genome quality/completeness assessment.
- Pharokka and PHANOTATE-style annotation approaches for phage annotation context.
- geNomad and VirSorter2 for broader mobile-element or viral-contig discovery ideas.
- Phold for structure-informed protein annotation artifacts.
- Clinker for synteny visualization.
- iPHoP for optional host-prediction evidence.
- vConTACT2 for network/taxonomy concepts if the project later needs larger-scale comparative classification.
- nf-core/phageannotator, What_the_Phage, and multiPhATE2 as pipeline-design references.

## v0.3 Acceptance Criteria

- Default lightweight runs remain fast and dependency-light.
- All heavy tools remain optional and disabled by default.
- Each optional module has a local executable/database contract.
- Each new output has report, package, and validator coverage.
- Summaries avoid raw sequence strings, private table dumps, and manuscript-grade interpretation.
- Documentation names limitations directly and separates software validation from biological conclusions.
