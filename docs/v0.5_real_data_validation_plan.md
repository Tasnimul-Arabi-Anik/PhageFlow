# PhageFlow v0.5 Real-Data Validation Plan

This plan defines how to demonstrate PhageFlow on public real-data examples for a future `v0.5.0` validation release. It is a software-validation plan only. It does not add datasets, accessions, biological conclusions, or new workflow modules.

Before running any public example, follow [`public_example_data_policy.md`](public_example_data_policy.md).

## Validation Principles

- Keep the default workflow lightweight and runnable without large databases.
- Use public inputs only when source, license or terms, citation, retrieval date, and checksum can be recorded.
- Keep downloaded inputs, databases, and generated results outside Git unless an intentionally tiny example passes the repository-size gate.
- Record the PhageFlow commit, command log, validator command, database versions when used, and interpretation boundary.
- Treat output as software evidence: expected files, tables, figures, manifests, and validator checks.
- Do not claim taxonomy, host range, infectivity, safety, therapeutic suitability, or manuscript-grade biological conclusions.

## Candidate Dataset Category 1: Complete Single Phage Genome

Purpose:

- Demonstrate that a public assembled genome can run through the minimal single-genome contract.
- Verify input QA, FASTA metrics, lightweight ORFs, codon summaries, report generation, manifests, figures, and validator behavior.

Input requirements:

- One public complete assembled phage genome FASTA.
- Stable source landing page or accession record.
- Source license or terms reviewed.
- Input SHA256 checksum and byte size recorded.

Suggested sample count:

- One genome.

Provenance fields:

```text
example_id
source_database_or_repository
accession_or_record_id
source_url_or_landing_page
retrieval_date
input_file_name
input_sha256
input_bytes
license_or_terms_checked
citation_or_reference
```

Commands to run:

```bash
bash bin/phageflow run \
  --input data/public_examples/<example_id>/<input_fasta> \
  --pangenome_method none \
  --outdir results/public_examples/<example_id>_single

bash bin/phageflow validate \
  --outdir results/public_examples/<example_id>_single
```

Expected output groups:

- `00_inputs/`
- `01_qc/fasta_stats/`
- `02_annotation/lightweight_orfs/`
- `03_codon_usage/`
- `99_report/`

Validator flags:

- No optional expectation flags for the minimal run.

Interpretation boundary:

- Confirms single-genome software execution and report completeness only.
- Does not validate genome classification, host range, biological function, or wet-lab relevance.

## Candidate Dataset Category 2: Small Related Phage Cohort

Purpose:

- Demonstrate cohort-mode execution on public assembled genomes.
- Verify duplicate checks, k-mer similarity, BLASTN intergenomic similarity, pangenome summaries, report tables, and figures.

Input requirements:

- A small public cohort of assembled phage genomes.
- A samplesheet with one row per genome.
- Source and checksum fields recorded for every input file.

Suggested sample count:

- Three to ten genomes.

Provenance fields:

```text
example_id
sample_id
source_database_or_repository
accession_or_record_id
source_url_or_landing_page
retrieval_date
input_file_name
input_sha256
input_bytes
license_or_terms_checked
citation_or_reference
```

Commands to run:

```bash
nextflow run main.nf \
  --input data/public_examples/<example_id>/samplesheet.tsv \
  --pangenome_method mmseqs \
  --outdir results/public_examples/<example_id>_cohort

bash bin/phageflow validate \
  --outdir results/public_examples/<example_id>_cohort \
  --require-pangenome-rows
```

Expected output groups:

- `00_inputs/`
- `01_qc/fasta_stats/`
- `02_annotation/lightweight_orfs/`
- `03_codon_usage/`
- `04_comparative/cohort_similarity/`
- `04_comparative/intergenomic_similarity/`
- `04_comparative/mmseqs_pangenome/`
- `99_report/`

Validator flags:

- `--require-pangenome-rows`

Interpretation boundary:

- Confirms cohort workflow execution, comparative table generation, pangenome artifacts, and report completeness.
- Does not claim taxonomic assignment, gene-family correctness, or biological relatedness beyond the selected computational metrics.

## Candidate Dataset Category 3: Host-Linked Phage/Host Example

Purpose:

- Demonstrate host-context reporting when host metadata or host FASTA inputs are explicitly supplied by the user.
- Verify host-context tables, host codon-adaptation artifacts when enabled, and report integration.

Input requirements:

- Public phage FASTA plus public host metadata or host sequence input.
- Clear source records for phage and host materials.
- A documented reason the example is used for software testing, not host-range proof.

Suggested sample count:

- One to three phage records with matching host-context inputs.

Provenance fields:

```text
example_id
phage_source_record
host_source_record
host_metadata_source
retrieval_date
phage_input_sha256
host_input_sha256_if_applicable
license_or_terms_checked
citation_or_reference
intended_validation_scope
```

Commands to run:

```bash
nextflow run main.nf \
  --input data/public_examples/<example_id>/phage_samplesheet.tsv \
  --host_samplesheet data/public_examples/<example_id>/host_samplesheet.tsv \
  --outdir results/public_examples/<example_id>_host_context

bash bin/phageflow validate \
  --outdir results/public_examples/<example_id>_host_context \
  --expect-host-adaptation
```

If only metadata context is supplied and host codon-adaptation artifacts are not expected, omit `--expect-host-adaptation`.

Expected output groups:

- `00_inputs/`
- `06_host_context/`
- `99_report/tables/host_context.tsv`
- `99_report/tables/host_codon_adaptation.tsv` when host adaptation is enabled and completed

Validator flags:

- `--expect-host-adaptation` only when the matching workflow inputs and outputs are present.
- `--expect-crispr-hits` only when CRISPR spacer matching was explicitly requested and completed.

Interpretation boundary:

- Confirms supplied-host context reporting only.
- Does not prove host range, host specificity, infection, or experimental compatibility.

## Candidate Dataset Category 4: Optional Heavy-Tool Example Subset

Purpose:

- Demonstrate that disabled-by-default heavy wrappers can run with explicit local executables and databases.
- Verify optional-tool output directories, software-version records, optional summary rows, optional metric rows, and strict validator flags.

Input requirements:

- Small public phage input subset.
- Installed optional executables.
- Explicit local database paths for database-backed tools.
- Database provenance recorded according to the public example-data policy.

Suggested sample count:

- One to three genomes.

Provenance fields:

```text
example_id
input_manifest_path
phageflow_commit
optional_tools_enabled
database_root
database_manifest_path
database_version_or_release_per_tool
run_command
validator_command
validation_outcome
```

Commands to run:

```bash
bash bin/phageflow install --with publication,structure,host-prediction,integrated

bash bin/phageflow db status --db-root "$PHAGEFLOW_DB_ROOT"
bash bin/phageflow db run-args --db-root "$PHAGEFLOW_DB_ROOT" --tools all --shell

nextflow run main.nf \
  --input data/public_examples/<example_id>/samplesheet.tsv \
  --run_trnascan true \
  --run_bacphlip true \
  --run_checkv true \
  --checkv_db /path/to/checkv_db \
  --run_abricate true \
  --run_pharokka true \
  --pharokka_db /path/to/pharokka_db \
  --run_genomad true \
  --genomad_db /path/to/genomad_db \
  --run_phold true \
  --phold_db /path/to/phold_db \
  --run_clinker true \
  --run_iphop true \
  --iphop_db /path/to/iphop_db \
  --run_phabox true \
  --phabox_db /path/to/phabox_db \
  --outdir results/public_examples/<example_id>_heavy_optional

bash bin/phageflow validate \
  --outdir results/public_examples/<example_id>_heavy_optional \
  --require-pangenome-rows \
  --expect-publication-optionals \
  --expect-phold \
  --expect-iphop \
  --expect-phabox
```

Expected output groups:

- `05_optional/`
- `99_report/tables/optional_tool_summary.tsv`
- `99_report/tables/optional_tool_metrics.tsv`
- `99_report/tables/functional_category_summary.tsv` when compatible annotation category tables are present
- `99_report/software_versions.tsv`

Validator flags:

- `--expect-publication-optionals`
- `--expect-phold` when Phold was enabled and completed
- `--expect-iphop` when iPHoP was enabled and completed
- `--expect-phabox` when PhaBOX/PhaBOX2 was enabled and completed

Interpretation boundary:

- Confirms optional wrapper execution and artifact capture with explicit databases.
- External-tool taxonomy, host-prediction, lifestyle, quality, and annotation outputs remain external-tool evidence and are not interpreted by PhageFlow.

## Release Evidence Template

For each completed public example, create a concise validation note:

```text
Example ID:
Validation date:
PhageFlow commit:
Input manifest:
Database manifest:
Run command:
Validator command:
Validation outcome:
Output package:
Known limitations:
Interpretation boundary:
```

The final `v0.5.0` validation summary should list which categories were completed, which were deferred, and why. A release can be described as real-data software validated only for the categories that have complete provenance and passing validator evidence.
